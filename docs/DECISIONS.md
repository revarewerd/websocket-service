## ADR-001: Smart Consumer вместо выделенного Kafka топика

**Статус:** Принято  
**Дата:** 2026-03-03  

**Контекст:** Рассматривались 3 варианта доставки GPS-данных через WebSocket:

| Вариант | Описание |
|---|---|
| A) Отдельный топик `gps-events-realtime` | CM публикует копию только для WS |
| B) Smart Consumer (выбран) | WS читает `gps-events`, фильтрует in-memory |
| C) Redis Pub/Sub | CM публикует в Redis канал, WS подписывается |

**Анализ:**

| Метрика | Вариант A | Вариант B | Вариант C |
|---|---|---|---|
| Нагрузка на CM | +3K writes/sec | 0 (никаких изменений) | +10K Redis pub/sec |
| Kafka трафик | +3K msg/sec (новый топик) | 0 (существующий топик) | 0 |
| Стоимость фильтрации в WS | ~0 | ~0.5ms HashMap lookup | ~0 |
| Изменения в CM | Да (новый producer) | Нет | Да (Redis publisher) |
| Масштабирование | Отлично | Хорошо (MVP) | Хорошо |

**Решение:** Вариант B (Smart Consumer). WS Service потребляет `gps-events` с `auto.offset.reset=latest`,
фильтрует точки через `ConnectionRegistry.getSubscribersForVehicle()` — это O(1) HashMap lookup.
Стоимость дискарда 7K msg/sec (не имеющих подписчиков) — ~0.5мс, что ничтожно.
Главное преимущество — **0 изменений в CM** и **0 дополнительного Kafka трафика**.

## ADR-002: Dual Index (vehicleIndex + orgIndex)

**Статус:** Принято  
**Дата:** 2026-03-03  

**Контекст:** Клиенты подписываются двумя способами:
1. На конкретные vehicleId (диспетчер следит за 5-10 машинами)
2. На всю организацию (экран "карта всех ТС")

Нужно при получении GPS точки быстро найти ВСЕ подписчиков — и тех, кто подписан
на конкретный vehicleId, и тех, кто подписан на всю организацию.

**Решение:** Два обратных индекса в ZIO Ref:
- `vehicleIndex: Map[Long, Set[UUID]]` — vehicleId → соединения
- `orgIndex: Map[Long, Set[UUID]]` — orgId → соединения с subscribeOrg

При роутинге: `vehicleIndex[vehicleId] ∪ orgIndex[orgId]` → полный набор подписчиков.

**Результат:** O(1) lookup для любого типа подписки. Не нужен scan по всем соединениям.

## ADR-003: In-Memory State (ZIO Ref) вместо Redis

**Статус:** Принято  
**Дата:** 2026-03-03  

**Контекст:** Для хранения подписок рассматривались:
1. Redis HASH/SET — позволяет горизонтальное масштабирование
2. ZIO Ref (in-memory) — проще, быстрее, но single-instance

**Решение:** ZIO Ref для MVP. Причины:
- Latency: ~0.001ms (memory) vs ~1ms (Redis)
- Нет дополнительной зависимости от Redis
- При 1000 WS соединений × 10 подписок = 10K записей — память ~1MB
- Горизонтальное масштабирование через Redis Pub/Sub — в post-MVP

**Риски:**
- При падении инстанса все подписки теряются → клиенты переподключаются
- При 2+ инстансах подписки не синхронизированы → нужен Redis Pub/Sub (post-MVP)

## ADR-004: PositionThrottle 1 sec per vehicleId

**Статус:** Принято  
**Дата:** 2026-03-03  

**Контекст:** GPS трекеры шлют данные каждые 1-60 секунд. При 10K трекеров с интервалом 1 сек — 
это 10K msg/sec. Отправлять каждое обновление в WebSocket:
- Нагрузка на клиент: браузер не успевает рендерить >1 FPS на карте
- Трафик: 10K × 200 байт = 2MB/sec на клиента (если subscribedToOrg)

**Решение:** `PositionThrottler` с `Ref[Map[Long, Instant]]`:
- На каждый vehicleId — максимум 1 обновление в `positionIntervalMs` (default: 1000мс)
- Бизнес-события (геозоны, скорость) — без throttle, доставка немедленная

**Конфигурируемость:** через `throttle.position-interval-ms` в application.conf.

## ADR-005: auto.offset.reset = latest

**Статус:** Принято  
**Дата:** 2026-03-03  

**Контекст:** При запуске WS Service должен ли он прочитать все непрочитанные сообщения 
(earliest) или только новые (latest)?

**Решение:** `latest`. Причины:
- WebSocket Service — real-time доставка, не обработка истории
- При рестарте все WS клиенты отключаются и переподключаются
- Старые GPS точки неактуальны — карта показывает ТЕКУЩУЮ позицию
- History Writer (with `earliest`) отвечает за полноту данных

**Следствие:** при рестарте WS Service будет лаг до первого GPS пакета от каждого трекера
(1-60 сек в зависимости от интервала).

## ADR-006: Единый порт для HTTP и WebSocket

**Статус:** Принято  
**Дата:** 2026-03-03  

**Контекст:** Можно ли разделить WS (:8090) и HTTP health (:8091)?

**Решение:** Один порт 8090 для обоих. zio-http поддерживает HTTP и WebSocket
на одном порту (WS — это HTTP Upgrade). Преимущества:
- Один порт → проще конфигурация Docker/K8s
- Один healthcheck endpoint для liveness probe
- K8s `Service` с одним target port

## ADR-007: orgId из Query Params (MVP) → JWT (post-MVP)

**Статус:** Принято (MVP)  
**Дата:** 2026-03-03  

**Контекст:** Как идентифицировать организацию клиента при WS подключении?

| Вариант | Безопасность | Сложность |
|---|---|---|
| Query param `?orgId=100` | Низкая (можно подставить чужой) | Минимальная |
| JWT токен в header | Высокая | Нужен Auth Service |
| JWT в query param | Средняя (токен в URL логах) | Нужен Auth Service |

**Решение MVP:** `?orgId=100` для быстрого старта без Auth Service.
**План post-MVP:** API Gateway валидирует JWT, прокидывает orgId в header `X-Organization-Id`,
WS Service доверяет ему как internal-only сервис.

**Риски MVP:** возможен доступ к данным чужой организации через подмену orgId.
Допустимо для dev/staging, критично для production.
